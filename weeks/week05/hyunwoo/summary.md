## 이번 주 핵심 요약

### 13장

- 리액트에서 Hook은 함수형 컴포넌트에서 리액트의 핵심 기능에 갈고리를 걸어 사용한다는 데에서 비유에서 유래
- 왜 클래스형 컴포넌트에서 함수형 컴포넌트로 넘어왔나?
- 첫번째 이유는 this의 문제. this는 호출되는 맥락에 따라 달라지는 경우가 많음. 그렇기에 bind 처리를 하거나 화살표 함수 등의 우회 필요.
- 클래스형 컴포넌트에서 this가 undefined 된 이유는 이벤트가 실행될 때는 "객체가 호출한 게 아니라 함수가 호출되는" 개념 때문. (아래 코드 참고)

```
const fn = instance.handleClick;
fn();  // 그냥 함수 호출
```

- 또한 고차 컴포넌트 지옥으로 인한 가독성 저해로 인해도 존재했음.
- 커스텀 훅 사용하면 상당부분 해결됨

- 훅은 최상위에서 호출되어야 함. 즉, 훅의 호출 순서 보장이 되어야 한다는 것이며 조건문 안에서 실행될 수 없음을 뜻함. 이유로는, 훅도 연결리스트로 연결되어 있기 때문에 조건부로 호출되어버리면 연결리스트 순서가 이상해져 치명적인 버그가 발생할 수 있기 때문임.
- 또한 리액트 함수 내에서만 훅 호출이 가능함. 리액트 함수 컴포넌트 최상위 혹은 커스텀 훅의 최상위 스코프에서 가능함. 렌더링 컨텍스트가 필요하기 때문임.
- 훅 내에서도 불변성을 유지해야 한다.

### 14장

- useState에서 지연 초기화를 사용하자. 컴포넌트는 매번 리렌더링될때마다 값이 재계산됨. 함수 인자를 넣어서 마운트에만 실행하고, 리렌더링에는 실행을 방지하는 전략

```
const [state, setState] = useState(() => initialValue);
```

- setState는 스케줄링된다. 당연히 다음 줄에 console.log 찍어도 업데이트 된 값이 반영되지 않는다.
- 업데이트 시에 최신값을 보장하고 싶다면 함수형으로 업데이트하고, 불변성을 보장하는 방식으로 변경하자.
- 원리는 리액트 상태 처리 방식에 있다. (409p)

```
setTasks(prevTasks => [...prevTasks, task])
```

- useEffect 사용법은 다들 아시니까 생략하겠습니다. 다만 <StrictMode>에선 리액트가 마운트되고, 시뮬레이션 목적으로 언마운트되고, 다시 마운트되는 점을 참고하여 당황하지 않아야 겠습니다. useEffect 특성상 컴포넌트 생애주기와 비교적 많이 연관되어 있기 때문입니다.

- useRef는 값이 변화해도 리렌더링을 발생시키지 않고, 값이 초기화되지도 않습니다. 그렇기에 렌더링과 관련된 값을 저장하기 좋고, 또한 DOM에 직접 접근이 가능해서 입력 필드에 포커스 두는 작업등을 할 수 있습니다.
- 또한 ref를 자식 컴포넌트에 넘겨주려면 forwardRef로 감싸줘야 하는데, React 19에서는 그냥 ref로 넘겨주면 되서 간편해졌습니다.
- useReducer는 상태 변화 로직이 복잡해진 경우 역할을 나눠주는 의미에서 사용하면 좋습니다. 다만 저는 필요한 경우가 없었습니다.
- createPortal은 컴포넌트 트리 구조를 뛰어넘어 직접 원하는 DOM 레벨에 위치시킬 수 있습니다. 컴포넌트 자체는 하위에 위치하지만 동작은 시각적으로는 상단으로 끌어올려져야 하는 모달, 토스트 컴포넌트 등에 사용됩니다.

## 15장

- 리액트 메모이제이션엔 memo, useMemo, useCallback이 존재합니다.
- memo는 함수형 컴포넌트 실행 자체를 막습니다. 기본적으로 이전 props, 현재 props를 돌아가며 Object.is()로 비교합니다. 두 번째 인자로 어떻게 비교할지에 대한 함수를 넘겨줄 수 있습니다.
- memo를 적용하게 되면 Fiber에서 bailout()을 시켜 더 이상 포인터가 자식으로 내려가지 않습니다. 다만, 무조건 감싼다고 되는 게 아니라 매번 생성되는 객체가 props로 넘어온다면 useCallback으로 감싸는 작업을 해야 작동합니다.
- memo의 성공적인 예시를 들겠습니다. 계층이 깊은 리스트 아이템 컴포넌트에서 리렌더링을 막으면, 불필요한 하위 자식들까지 리렌더링되지 않아 성능 이점을 누릴 수 있는 것입니다. 반면에 책에 나온대로 breadCrumb와 같은 단순한 컴포넌트는 비교하는 비용이 더 크게 들어 불필요합니다.
- children props로 memo를 사용한 Child 컴포넌트를 넘겨주면 자식은 처음 1회만 렌더링되지만, 부모는 매번 리렌더링됩니다. 결국 내부적으로 createElement()를 호출해 새 객체를 만들어내기 때문입니다. 이런 경우 useMemo를 활용하는 방식으로 해결이 가능합니다. 즉, createElement의 결과물도 메모이제이션 하는 것입니다.

- useMemo는 값을 메모이제이션하고 첫번째로는 연산 결과를 반환하는 함수, 두번째로는 의존성 배열을 받습니다. 의존성 배열에 명시된 값이 변할때마다 연산 결과를 새로 저장합니다.
- 당연히, 복잡한 계산식에 사용됩니다. 최초 1회만 실행되면 되거나, 특정 상태가 변경된 경우만 재계산되면 되는 경우에 사용됩니다.

- useCallback은 함수를 메모이제이션하고 원리는 같습니다. 당연히 컴포넌트가 리렌더링되면 함수도 리렌더링되어 예상치 못한 결과가 발생할 수 있습니다. 463p과 같이 매번 새 함수가 생성되어 예상치 못한 동작이 발생 가능하고, 이 때 useCallback을 사용하여 최초 생성된 함수를 계속 사용할 수 있습니다.

- 이러한 것들을 수동으로 하다 보니 코드 복잡성 증가, 실수하기 쉬움, 유지보수의 어려움등과 같은 문제가 발생하였습니다. 그래서 리액트 컴파일러라는 개념이 들어왔습니다.
- 리액트 컴파일러는 위에서 설명한 memo, useMemo, useCallback을 사용하지 않아도 알아서 최적화를 해 줍니다. 전체적인 코드의 양이 감소되어 가독성 향상을 기대합니다.
- 리액트 컴파일러는 컴포넌트가 멱등적이며, 순수 함수이며, 리액트 훅의 규칙을 준수한 것이 확인되야 최적화를 진행합니다.
- 리액트 컴파일러가 충분히 좋은 최적화를 해 준다고 해도, 컴포넌트의 맥락은 작성자만 알 수 있습니다. 최고의 최적화를 하기 위해선 개발자가 맥락을 파악하고, 판단할 수 있어야 합니다. (471p)
- 리액틐 컴파일러를 사용하려면 babel-react-compiler를 설치하고 설정을 해 주면 됩니다. (474p, vite.config.ts) 적용 범위도 제한할 수 있습니다.
