## 13. 리액트 훅의 사용 규칙 돌아보기
### hook이란?
- 클래스 컴포넌트 → 함수형 컴포넌트
  - 클래스 컴포넌트의 복잡성을 훅과 함수형 컴포넌트가 어떻게 해결했는지
- JS 함수였던 컴포넌트가 React의 상태를 직접 소유하고, 컴포넌트의 마운트, 업데이트, 언마운트와 같은 생명 주기 시점에 맞추어 특정 로직을 실행할 수 있게 해줌
- ex. useState(), useEffect()

### 클래스 컴포넌트의 단점
#### this 키워드의 혼란과 수동 바인딩의 번거로움
- this
  - 컨텍스트 (함수가 호출되는 방식)에 따라 동적으로 결졍
  - 클래스 컴포넌트에서 예측하기 어려운 버그의 원인인 됨 (ex. undefined가 되어버린 this)

#### HOC 패턴
- 컴포넌트를 인자로 받아서, 새로운 컴포넌트를 반환하는 함수
- 래퍼 지옥 (Wrapper hell) 발생
  - 클래스 컴포넌트에서 상태 관련 로직이나 생명주기 로직을 여러 컴포넌트에 재사용하는 방법
  - 재사용성 ↑ BUT 컴포넌트를 여러개의 다른 컴포넌트로 감싸는 구조 생성
    ```tsx
    export default withAuth(withTheme(withLogging(Page)))
    ```

**→ 위와 같은 클래스 컴포넌크와 HOC 패턴의 복잡성으로 등장한 것이 함수형 컴포넌트 & 훅**

### 커스텀 훅
- 'use'로 시작되는 이름의 js 함수
- 컴포넌트의 상태 관련 로직을 캡슐화하고 재사용할 수 있게 해줌

### React Hook 사용 규칙
#### 1.  훅은 **최상위에서만 호출** 되어야 함
- 컴포넌트가 렌더링될때마다, 훅은 항상 동일한 순서로 호출되어야함
  - 반복문, 조건문, 중첩된 함수 내에서 호출 금지
- React는 컴포넌트별로 **훅의 상태를 저장하는데 단일 연결 링크드리스트**를 사용
- 그리고 훅이 호출되는 순서, **링크드 리스트를 순서대로 순회**하며 각 훅의 상태를 가져오고 상태를 구분
- **컴포넌트가 리렌더링될때마다, 링크드 리스트를 순서대로 순회하며 기존에 저장해둔 노드에서 각 상태와 상태 업데이트 함수를 참조하기 때문에**! 훅의 순서를 반드시 지켜야한다는 것

#### 2. 오직 React 함수 내에서만 Hook을 호출해야함
- 리액트 훅이 호출될 수 있는 위치
  - React 함수 컴포넌트의 최상위 스코프
  - 커스텀 훅의 최상위 스코프
- **현재 어떤 컴포넌트를 렌더링하고 있는지**에 대한 정보인 **렌더링 컨텍스트**가 훅 동작에 필요하기 때문

#### 3. 훅의 인수는 불변성을 가지고 있어야 함
- 얕은 비교를 하기 때문에, 변경이 필요하다면 원본 직접 수정이 X
-  [...] 스프레드 연산자등을 사용해 복사본을 만들어 변경 사항을 적용한 새로운 객체 / 배열을 반환해야함

## 14. 리액트 필수 훅 돌아보기
### useState()
#### 상태 초기화와 지연 초기화
- 함수형 컴포넌트에 지역 상태를 추가하는 기본 훅 (리액트 상태 관리의 시작)
- setState() 호출 직후에 상태를 읽으면 이전 값이 보임!
  - React는 배치 업데이트를 수행하니까 (상태 업데이트 스케줄링 → 여러개 묶어서 처리)

#### 상태 업데이트와 업데이트 함수
- So, 이전 상태에 의존하여 다음 상태를 계산해야할 때에는 **함수형 업데이트** 사용
  - setState()에 값을 직접 전달하는 게 X라, 콜백 함수를 전달하는 방식
  - closure에 의존하지 x
  - setState(prevState => newState) 에서 preState는 항상 최신 상태임을 React가 보장

#### Stale closure
- **Closure** : 변수가 만들어질 때의 “변수 환경”을 기억하는 것
  - 각 상탯값은 각 파이버 노드의 **memoized State**에 저장
- **단계**
  - useState는 값을 “변경”하는 게 X, 새로운 렌더를 만들고 그 렌더 안에 새로운 값을 넣는 것
  - closure는 자신이 만들어진 “그 렌더의 값”을 기억 → 여전히 이전 값을 바라보고 있는 것
- **예시**
  - 버튼을 누른다
  - state가 바뀐다
  - 새로운 렌더가 만들어진다
  - BUT 예전 렌더에서 만들어진 함수는 → 여전히 예전 값을 보고 있다 (Stale closure)
- **해결책**
  - 함수형 업데이트 (setState(prev => ...))
  - useRef
  - 의존성 배열 관리

### useEffect()
- 컴포넌트를 외부 시스템과 연결하고 동기화하는 역할

#### useEffect의 의존성 배열
- Effect 안에서 외부 함수를 호출한다면 → 넣어야 함
  - **딘** 해당 함수가 **useCallback으로 메모이제이션**된 경우에만 → 넣어도 괜찮음
  - 그렇지 않다면, 함수는 매 렌더링마다 새로운 객체로 생성되니까 무한 실행 가능해짐

### useRef()
- 렌더링과 관계없이 특정 값을 컴포넌트의 전체 생명주기에 걸쳐 유지하고 싶은 경우
  - 함수형 컴포넌트는 렌더링 될때마다 내부의 변수가 초기화됨

#### 리렌더링을 발생시키지 X는 useRef()
- ref.current 프로퍼티를 변경해도 react가 리렌더링을 발생시키지 X
- **비교**
  - **useState** : 값의 변경을 UI에 반영해야하는 경우 (리렌더링 발생하니까)
  - **useRef** : 렌더링과 무관하게 값을 보관해야하는 경우 (리렌더링 발생 X하니까)

#### DOM 요소에 직접 접근하여 메서드를 호출하는 경우
- ex. 스크롤 위치 제어하기, 입력 필드에 자동으로 포커스 맞추기
- DOM 요소에 접근하는 로직은 반드시 useEffect() 내부에서 수행되어야함
  - 렌더링 과정 중에는 ref.current가 아직 null일 수 있기 때문

```tsx
function SearchInput() {
  // 1. 객체 생성
  const inputRef = useRef<HTMLInputElement>(null);

  useEffect(() => {
    // 3. useEffect는 컴포넌트가 렌더링되고, ref가 연결된 후 실행
    // inputRef.current가 실제 <input> DOM 노드를 가리킴
    inputRef.current?.focus();
  }, []);

  // 2. inputRef와 실제 DOM 노드 연결
  return <input ref={inputRef} type="text" placeholder="검색어 입력" />;
}
```

### forwardRef()
#### **React 18까지** → 일반적인 props처럼 ref를 컴포넌트에 직접 전달 불가능
- 컴포넌트 간에 ref를 전달하기 위해서 **forwardRef()는 필수적**
- 부모 컴포넌트로부터 전달된 ref를 받아서, 자식의 특정 JSX 요소에 연결해주는 함수
- 넘겨받는 자식 컴포넌트에서 forwardRef를 사용하여 받음
  ```tsx
  const Comp = forwardRef((props, ref) => {})
  ```

#### **React 19부터** → ref를 일반 프롭스부터 사용할 수 있는 기능 생김
- forwardRef 없이 ref props 전달 가능해짐 (이제 불필요해진 forwardRef)
  ```tsx
  const Comp = ({props, ref }) => {}
  ```

## 15. 리액트 메모이제이션 돌아보기
- 메모이제이션은 **값을 비교하고, 이전 값을 기억하기 위한 메모리와 연산 비용**이 생김 <br/>
→ 해당 비용을 넘는 지점에서 사용해지만 **최적화의 이점**이 발생

### memo( )
- 컴포넌트의 렌더링 결과를 메모이제이션

- **기본 규칙** : 부모가 리렌더링되면 자식도 리렌더링 됨
- **memo( )** : 자식 컴포넌트의 props가 이전과 동일하다면 리렌더링 X말고, 이전 결과를 재사용하자!

#### memo( ) 를 사용하는 것이 좋은 경우
- 컴포넌트가 순수하고, 렌더링 비용이 비싼 경우
  - 복잡한 계산을 하거나, 무거운 UI 요소를 렌더링 하는 경우
- 컴포넌트가 동일한 props로 자주 렌더링되는 경우
  - 부모의 상태 변화 때문에 상관없지만 자식 컴포넌트가 자주 리렌더링 되는 경우
- props가 복잡한 객체를 포함한 경우
  - 함수나 객체등의 props를 받는다면, useCallback(), useMemo()를 사용해야함
  - memo()는 컴포넌트의 렌더링 결과만 메모이제이션 / 전달되는 props를 안정화해주지는 X기 때문

### useMemo()
- 컴포넌트 내부에서 사용하는 특정 값 / 연산의 결과를 메모이제이션
- 의존성이 같다면, 이전 렌더링에서 계산된 값 그대로 사용

### useCallback()
- 함수 자체를 메모이제이션
- 메모이제이션 수행할 함수, 의존성 배열을 전달

#### memo()와 useCallback()
- memo()로 자식 컴포넌트를 최적화해도, 부모가 자식에게 함수를 props로 전달하면 최적화가 깨지는 경우 多 <br/>
→ props로 전달되는 함수도 **useCallback()**을 사용해서 메모이제이션해야함

#### debounce( ) 함수
- 특정 함수 호출 사이에 지연 시간을 두어, 이벤트가 연속으로 빠르게 발생할때 함수가 과도하게 실행되는 것을 방지
- **동작 원리**
  - 함수가 호출되면, 지정된 시간만큼 타이머를 설정 (ex. 300ms)
  - 설정된 시간 내에 같은 함수가 재호출되면, 기존 타이머를 취소하고 새로운 타이머를 설정
  - 타이머 만료전까지 추가 호출 X면, 마지막으로 예약된 함수 로직 실행
- ex. 유저가 검색 창에 텍스트를 입력하는 동안 키 입력마다 API를 호출하는 대신, 입력이 잠시 멈췄을때 한번만 API를 호출할 수 있도록 제어

- cf) insight.md 에 추가 설명

### 리액트 컴파일러
- **등장 배경**
  - 수동 메모이제이션의 한계 (useMemo, useCallback, memo 등)
    - 코드 복잡성 증가, 실수 쉬움, 유지보수 어려움
- **발표**
  - react 19와 함께 발표되었지만, 아직 출시되지는 X
  - 현재는 RC 버전으로만 사용 가능

### 리액트 컴파일러의 역할
- 자동 메모이제이션
- Cascading 렌더링 방지
  - 리액트의 기본 동작 : 부모가 리렌더링 되면 자식 컴포넌트도 리렌더링 되는 현상 = Cascading 렌더링
  - props나 상태가 변경될때만 자식이 리렌더링되게 변경될 예정
- 코드 가독성 향상
  - useMemo, useCallback등의 사용이 대부분 사라짐
