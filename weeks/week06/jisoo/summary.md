## 16. 리액트 컨텍스트와 에러바운더리, 그리고 서스펜스 돌아보기

### 에러 바운더리
- 방법 1) 클래스 컴포넌트 사용하여 구현
- 방법 2) react-error-boundary 라이브러리 사용

#### 1) 클래스 컴포넌트 사용하여 에러 바운더리 컴포넌트 만들기
- getDerivedStateFromError() 정적 생명주기 메서드 : 에러를 포착하여 폴백 UI 렌더링
- componentDidCatch() 생명주기 메서드 : 포착된 에러 정보를 기록 (센트리 / 데이터독에 기록)

- **에러 전파**
  - 컴포넌트 트리를 따라 상위로 전파 (에러 발생 지점부터 시작해서 가장 가까운 부모 에러 바운더리를 찾을때까지 위로 올림)
  - 에러 바운더리를 만나면, 에러 전파는 그 지점에서 멈춤

#### 2) react-error-boundary 라이브러리
- 에러 발생 시 보여줄 UI를 렌더링 함수 || 컴포넌트 형태로 전달할 수 있는 FallbackComponent와 같은 프롭스 제공
- useErrorHandler() 훅 제공 
  - 이벤트 핸들러 등에서 발생한 에러를 가장 가까운 에러 바운더리로 전달

### Context API
- 구체적인 로직 구현은 부모에게 감추면서 무엇을 렌더링할지 제어권은 부모에게 부여하는 것
- useData() 훅을 통해 곧바로 상태에 직접 접근 가능
- **전역 상태 도구** 가 X라 **프롭스 드릴링을 피하기 위해 만든 도구**
  - 주요 상태 관리 전략으로 사용할때 **리렌더링 문제 + 기타 주의접들** 고려 필요

#### Context API를 주요 상태 관리 전략으로 사용했을때 문제 + 주의점
1. 컨텍스트 제공자의 value props 메모이제이션
- 컨텍스트 제공자를 포함하는 상위 컴포넌트가 리렌더링 될때, value props로 전달되는 객체나 배열이 매번 새롭게 생성되면 구독하는 모든 하위 컴포넌트들이 불필요하게 리렌더링될 수 있음
  - 객체의 참조값이 달라지니까
- useMemo()를 contextValue에 사용해서 해결 가능

2. 컨텍스트 분리로 리렌더링 영향 범위 줄이기
- 컨텍스트를 구독하는 컴포넌트는 선택적 구독이 불가능! (셀렉터 패턴 제공 X)
  - 관심사별로 컨텍스트를 분리하자

### Suspense 
- 비동기 작업이 완료될때까지 UI 렌더링을 일시 중단하고, 로딩 폴백 UI를 보여주는 것
- React 19부터 어떠한 컴포넌트가 \<Suspense\>에 의해 일시 중단되면 즉시 가까운 경계의 폴백 UI를 먼저 커밋하고 형제 컴포넌트 전체 트리가 렌더링되기를 기다리지 X
  - 더 빠르게 폴백 UI를 표시할 수 있게 됨

#### Suspense 작동 원리
1. Promise 객체 throw
- 필요 데이터나 코드가 준비 되지 않은 경우, 렌더링을 멈추고 throw

2. catch로 Promise를 잡는 리액트
- try catch문으로 잡음

3. 폴백 UI 렌더링 - Wip 트리 조작
- 화면에 보이는 현재 상태 트리를 복사해서 새로운 UI를 그릴 WIP 트리를 준비
- WIP 트리에서 \<Suspense\> 파이버 노드의 child를 props.fallback에 있는 요소로 교체
- 원래 렌더링 해야하는 요소는 서스펜스 파이버 노드의 PendingProps.children 등에 저장
- 폴백 UI를 가진 WIP 트리를 커밋해서 실제 화면에 반영

4. Push 방식의 리스너를 통한 컴포넌트 렌더링 재개 예약

5. 실제 콘텐츠 렌더링 - 두번쨰 커밋
- Current 트리 복사해서 새로운 WIP 트리 생성
- 원래 자식 복원
- WIP 트리가 커밋되어 실제 데이터가 화면에 표시됨

### use() 훅
- 리액트 19에서 등장
- **반복문이나 조건문**안에서도 사용 가능하다는 특징
- 비동기 작업을 다룰때, 데이터가 아직 준비되지 않았다면 Suspense를 통해 선언적으로 로딩 상태를 관리하고, 데이터가 준비되면 동기 코드와 같이 값을 반환해주는 역할
- use()와 \<Suspense\>를 같이 사용하면 **데이터를 가져오면서 동시에 렌더링 가능** 

#### useContext()에 비해 use()가 가지는 이점
- useContext() 최상위 레벨 호출 규칙으로부터 자유로움
  - use() API는 조건문이나 반복문 내에서도 호출이 가능

#### use()가 조건문 안에서 호출될 수 있는 이유
- 다른 훅과는 다르게 호출될 떄 리액트의 내부적인 훅 리스트에 의존하여 상태를 저장하거나 검색하지 X
  - 훅들의 연결 리스트 구조와는 무관하게 동작
- 프로미스 객체 자체나 파이버 트리를 직접 탐색하는 독립적인 메커니즘을 통해 값을 읽음
  - 훅들의 상대적인 순서나 인덱스와 같은 위치와는 무관하게 독립적으로 동작


## 17. 리액트 동시성 기능과 심화 훅 돌아보기

### useLayoutEffect()
- Dom 변경 후 화면에 그려지기 직전에 동기적으로 실행
  - useLayoutEffect() 내부의 코드가 완료될 때까지 브라우저는 화면을 그리지 X
  - 시각적인 깜빡임 완화는 가능 BUT 페이지 로딩 시간 자체가 길어질 수 있다는 문제점 있음

### useInsertionEffect()과 리액트 헬멧

#### React Helmet
- \<head\> 내부에 위치되어야할 title, style, meta, script등을 작성하는 라이브러리
- 컴포넌트 내부에서도 HTML의 메타데이터 간편 변경 가능

#### useInsertionEffect()
- 리액트 18버전에서 추가
- 브라우저가 DOM 변경 사항을 커밋하기 전에 동기적으로 실행

### 리액트 19 버전의 메타 데이터
- 별도의 라이브러리를 사용하지 X아도 브라우저에서 기본적으로 사용할 수 있는 엘리먼트를 통해 페이지의 head 내부에 적용하고 싶은 title, link, script 등을 어디서든 선언할 수 있음


### 명령형 상호 작용이 필요한 경우 사용 가능한 훅
- 부모 컴포넌트가 자식 컴포넌트의 특정 동작을 직접적으로 제어하거나 자식 컴포넌트의 특정 내부 기능을 호출하는 경우가 발생
  - cf. 리액트의 기본 원칙 : 부모에서 자식으로 데이터를 전달하는 단방향 흐름 <- 해당 원칙에 예외적인 행동을 진행하는 것

#### useImperativeHandle()
- 부모가 선언적이 아니라 명령형으로 자식 컴포넌트의 동작을 직접 제어할 수 있게 해줌 = 제어의 역전
- 자식 컴포넌트는 부모가 ref를 통해 호출할 수 있는 함수들 = 명령형 핸들 을 직접 정의 가능
- ex. 자식 컴포넌트의 리셋 함수나 특정 액션을 수행하는 메서드를 호출해야할 때

### 동시성 기능
- **타임 슬라이싱**
  - 메인 스레드에 제어권을 양보함으로써, 유저 입력 처리와 같이 현재 실행 중인 작업보다 더 높은 우선순위를 가진 다른 작업이 먼저 처리되는 것
- 리액트의 동시성 특징과 타임 슬라이싱 매커니즘을 개발자가 직접 활용하여 UI 업데이트의 우선순위를 관리
- createRoot()를 사용하면 동시성 모드를 지원하게 되어 동시성 기능 사용 가능

#### useTransition()
- 특정 상태 업데이트를 transition으로 표시하여 리액트에게 해당 업데이트가 긴급하지 않음을 알리는 것
  - 리액트는 다른 업데이트 먼저 처리하고, 트랜젝션으로 표시된 작업들은 백그라운드에서 처리

#### useDerredValue()
- 특정 값의 지연된 버전을 생성하여, 해당 값을 사용하는 부분의 업데이트가 긴급한 다른 렌더링을 방해하지 않도록 함
  - 무거운 렌더링이 긴급한 유저 인터렉션을 차단하지 않도록 함으로써, 체감 성능과 반응성 향상

#### startTransition()
- 리액트 19부터는 startTransition에 비동기 함수를 전달 가능해짐
- isPending 상태 1개만으로 전체 비동기 과정의 로딩 UI를 일관되게 표시 가능


#### useSyncExternalStore()
- 편리하게 외부 상태를 이용하여 컴포넌트의 반응성을 이끌어낼 수 있음
- **티어링**
  - 동시성 기능 환경에서 여러 컴포넌트가 같은 외부 데이터 소스를 참조할때 각기 다른 버전의 데이터를 읽어 UI 일관성이 없게 보이는 현상
