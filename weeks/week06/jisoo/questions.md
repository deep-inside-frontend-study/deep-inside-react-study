## 에러바운더리 처리 
- Error Boundary를 서비스 전체에 하나만 두는 것이 좋을까, 기능별로 쪼개는 것이 좋을지 고민입니다. 실제로 어떻게 에러바운더리 처리를 하고 계신지 궁금합니다.
- 범위가 너무 넓으면 작은 에러에도 전체 화면이 화이트아웃(또는 폴백) 되는 현상 있을 듯

### Error boundary 개선 시도
#### 기존
- 흰 화면에 에러 메시지 문구 뜸

#### 개선하고자 한 방향
- 에러 발생 제외 하면 (최소 Layout은 남기고) api 코드에 맞는 에러 / 프론트 에러 메시지를 알럿 안에 띄우자
- API Intercepter와 Error Boundary 사용

#### 문제점
- 에러 처리의 주권이 파편화되어있었음
    - Axios Interceptor가 잡는 에러, 개별 try-catch가 잡는 에러, 그리고 아무도 잡지 않아 Error Boundary까지 흘러가는 에러 등등..
    - 이 상황에서 Error Boundary를 도입하니 일관성이 깨짐
        - 어떤 에러는 바운더리에 잡혀서 폴백 UI가 뜸
        - 어떤 에러는 중간에 catch 문이 가로채서 알럿만 뜨고 화면은 깨진 채로 남음
        - 기존에 떠야하는 에러 메시지 대신 Api intercepter에서 처리한 에러가 뜸
- '가짜 200(Pseudo-200)' 에러. 
  - HTTP 상태 코드는 성공(200 OK)인데, 실제 비즈니스 로직 응답값(data.success: false)은 실패인 경우
    - 해당 케이스에서 에러 메시지를 별도로 띄우는 경우가 이미 있었음
- 실제 error 코드로 내려오는 경우에서도 구좌마다 에러 메시지를 처리해둔 부분이 있었음
- 결국 모든 API 응답의 규격을 맞추고 에러 전파 경로를 단일화하기 전까지는 불가능하다고 생각하고 롤백..
