- `use()`와 `Suspense`를 활용하면 기존의 `useEffect + useState` 기반 데이터 페칭 패턴보다 코드가 훨씬 선언적으로 보이는데, 실제 실무에서도 모든 비동기 데이터를 이 방식으로 옮기는 것이 맞을까? 아니라면 어떤 경우에는 기존 패턴이나 React Query 같은 라이브러리 방식이 더 적절할까?

- `useTransition`과 `useDeferredValue`는 둘 다 사용자 경험을 부드럽게 만들기 위한 도구인데, 둘의 차이를 "상태 업데이트의 우선순위를 낮추는 것"과 "값의 반영을 늦추는 것"으로 이해해도 충분할까? 실제 검색 UI나 탭 전환 UI를 만든다면 각각 어느 상황에서 더 적절할지 궁금하다.

- `ErrorBoundary`, `Suspense`, `use()`를 함께 보면 React가 로딩과 에러를 렌더링 경계 단위로 다루려는 방향이 분명해 보인다. 그렇다면 앞으로의 React 애플리케이션에서는 비즈니스 로직 단위보다도 UI 경계를 어떻게 나누는지가 더 중요한 설계 포인트가 될까?

- `useSyncExternalStore`는 외부 상태를 React와 안전하게 연결하기 위한 표준 API라고 이해했는데, 그렇다면 Zustand, Redux 같은 상태관리 라이브러리는 내부적으로 이 훅을 얼마나 활용하고 있을까? 그리고 이 훅이 생긴 이후로 "React 바깥 상태"를 다루는 방식이 어떻게 정리되었는지도 궁금하다.
