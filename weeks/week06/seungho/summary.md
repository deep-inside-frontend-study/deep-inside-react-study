# chapter 16. 리액트 컨텍스트와 에러 바운더리, 그리고 서스팬스 돌아보기

### 에러 바운더리 만들어보기

- 클래스 컴포넌트를 사용해서 에러 바운더리를 만들 수 있다.
- 두 메서드가 존재한다.
  - static getDerivedStateFromError(error): 렌더링 과정에서 에러가 발생했을 때 호출된다. 에러를 인자로 받는만큼 컴포넌트의 상태를 갱신할 수 있다. 상태 갱신시 컴포넌트 리렌더링이 되고, 이 때 개발자가 준비한 대체(Fallback) UI를 보여줄 수 있다.
  - componentDidCatch(error, errorInfo): 에러가 포창된 후 호출된다. 주로 센트리(Sentry)나 데이터독(Datadog) 같은 외부 에러 모니터링 서비스에 에러 정보를 전송하는 등의 부수 효과를 처리하는 데 사용된다.

### 에러전파 알아보기

- 랜더링 과정에서 에러가 발생하면 컴포넌트 트리를 따라 **상위로** 전파된다.
- 에러 발생 지점부터 가장 가까운 부모 에러 바운더리를 찾아가는 과정을 에러 전파라고 부른다.
- 당연히 에러 바운더리에서 막히면 더 이상은 전파되지 않는다.
- 에러전파 조건은 리액트 렌더링 생명주기, 재조정 동안 발생하는 에러만 감지할 수 있다. 이벤트 핸들러 내 에러나 fetch 비동기 에러를 잡을 수 없다는 이야기다.
- 그럼 사용이 너무 제한적이라고 생각할 수 있지만, throw error를 통해 명시적으로 에러를 발생시킨다면 에러 바운더리를 활용할 수 있다.

```js
useEffect(() => {
  throw new Error('에러 메시지');
}, []);
```

### Context API 사용할 때 주의해야 될 점

- provider의 value가 변경되면, 하위 컴포넌트들이 모두 리렌더링된다. 따라서 value 객체에 참조값이 달리면 컨텍스트를 구독하는 모든 컴포넌트들이 불필요하게 리렌더링된다. 이럴때는 useMemo hook을 사용해서 value props를 메모이제이션해 최적화할 수 있다.

```js
const value = useMemo(() => ({ name: 'John Doe' }), [name]);

return (
  <provider value={value}>
    <Child />
  </provider>
);
```

- Context를 적절히 분리하는것도 중요하다. 컨텍스트를 구독하는 컴포넌트들이 있는 곳에서만 컨텍스트를 제공하고, 그 외에는 제공하지 않는 방식으로 분리할 수 있다. 그리고 관심사 별로 컨텍스트를 분리해서 사용하는 것이 좋다.

### 서스펜스 알아보기

- 서스펜스는 비동기 작업이 완료될 때까지 UI 렌더링을 선언적으로 일시 중단하고, 이 기간동안 Fallback UI를 보여주는 매커니즘이다.
- 기본적으로 우리는 기존에 주로 React.lazy() API를 사용하며 경험해 보았을 것이다.
- 조금 더 내부 동작 관점에서 보면, 렌더링 중 아직 준비되지 않은 값이 있으면 React는 해당 렌더링을 잠시 멈추고 가장 가까운 `<Suspense>`의 fallback을 보여준다. 그리고 비동기 작업이 끝나면 해당 트리를 다시 렌더링한다.
- 예전에는 `wrapPromise` 같은 패턴으로 Suspense를 설명하곤 했다. 핵심은 간단하다.
  - Promise가 `pending` 상태면 Promise를 `throw`해서 `<Suspense>`가 fallback을 보여주게 만든다.
  - Promise가 `rejected` 상태면 error를 `throw`해서 `<ErrorBoundary>`가 처리하게 만든다.
  - Promise가 `fulfilled` 상태면 실제 데이터를 반환해서 컴포넌트가 그대로 렌더링된다.
- 즉 Suspense는 "로딩 상태를 어디서 어떻게 보여줄지"를 선언적으로 다루는 도구이고, 에러 처리는 보통 `<ErrorBoundary>`와 함께 조합해서 생각해야 한다.
- 그래서 실무에서도 `<ErrorBoundary>` 바깥에 `<Suspense>`를 두거나, 반대로 `<Suspense>` 바깥에 `<ErrorBoundary>`를 두는 식으로 로딩과 에러 UI를 분리해 설계한다.

### React 19의 use() 알아보기

- React 19에서 추가된 `use()`는 Promise나 Context 같은 리소스의 값을 읽기 위한 API다.
- Promise를 `use()`에 넘기면, Promise가 아직 끝나지 않았을 때 React가 자동으로 suspend하고 가장 가까운 `<Suspense>`의 fallback을 보여준다. Promise가 resolve되면 그 값을 반환하고, reject되면 가장 가까운 `<ErrorBoundary>`로 에러가 전파된다.
- 이 점에서 `use()`는 기존의 "Promise를 직접 throw해서 Suspense에 연결하던 패턴"을 조금 더 React스럽고 선언적인 API로 감싼 형태라고 이해하면 쉽다.
- Context도 읽을 수 있다. `useContext()`와 비슷하지만, `use()`는 일반적인 Hook과 다르게 조건문이나 반복문 내부에서도 호출할 수 있다는 차이가 있다.
- 다만 아무 데서나 호출할 수 있는 것은 아니다. 여전히 React 컴포넌트나 커스텀 Hook 내부에서만 사용해야 한다.
- 특히 Promise를 다룰 때 장점이 크다. 우리가 직접 `isLoading`, `error`, `data` 상태를 나눠 관리하지 않아도 되고, 로딩은 `<Suspense>`, 에러는 `<ErrorBoundary>`, 정상 결과는 `use()`가 반환한 값으로 자연스럽게 분리할 수 있다.
- 그래서 `useEffect` 안에서 fetch하고, `useState`로 loading/error/data를 따로 관리하던 패턴보다 코드가 더 선언적이고 읽기 쉬워진다.
- 다만 렌더링 중에 매번 새 Promise를 만들면 재렌더링 때마다 다른 Promise가 생길 수 있으므로 주의해야 한다. 가능하면 안정적인 Promise를 전달하거나, 서버 컴포넌트에서 Promise를 만든 뒤 클라이언트 컴포넌트로 넘겨 `use()`로 읽는 방식이 더 적합하다.
- 그리고 `use()`는 `try/catch`와 함께 쓰는 방식이 아니라 `<ErrorBoundary>` 또는 `Promise.catch()`와 함께 설계해야 한다는 점도 기억해두면 좋다.

### 17장

- `useLayoutEffect`
  - `useEffect`와 비슷하지만, 브라우저가 화면을 다시 그리기 전에 실행된다는 점이 핵심이다.
  - 그래서 툴팁 위치 계산, DOM 크기 측정처럼 "화면이 보이기 전에 끝나야 하는 작업"에 적합하다.
  - 대신 이 훅 안의 로직은 페인팅을 막을 수 있다. 즉, 잘못 쓰면 화면 깜빡임은 줄일 수 있어도 전체 체감 성능은 나빠질 수 있다.
  - 결론적으로 레이아웃 측정이 꼭 필요한 경우에만 쓰고, 대부분의 동기화 작업은 `useEffect`로 해결하는 편이 안전하다.

- 메타데이터 관리
  - 예전에는 페이지별 `<title>`, `<meta>` 같은 문서 메타데이터를 관리하려고 `react-helmet` 같은 라이브러리를 많이 사용했다.
  - React 19에서는 이런 태그를 컴포넌트 안에서 직접 렌더링해도 React가 알아서 문서의 `<head>` 쪽으로 올려준다.
  - 그래서 라우트나 페이지 컴포넌트 가까운 곳에서 메타데이터를 선언할 수 있게 되었고, 스트리밍 SSR이나 서버 컴포넌트 환경에서도 더 자연스럽게 동작한다.

- `useInsertionEffect`
  - 이 훅은 일반적인 앱 개발자가 자주 쓸 훅이라기보다는 CSS-in-JS 라이브러리 작성자에게 더 가깝다.
  - 목적은 다른 layout effect가 실행되기 전에 스타일을 먼저 주입하는 것이다.
  - 스타일 태그가 늦게 들어가면 잘못된 레이아웃 계산이나 불필요한 스타일 재계산이 발생할 수 있는데, `useInsertionEffect`는 이런 문제를 줄이기 위한 타이밍용 훅이다.
  - 공식 문서도 대부분의 경우 `useEffect`나 `useLayoutEffect`를 먼저 고려하라고 안내한다.

- 스타일시트 처리
  - React 19에서는 외부 스타일시트도 컴포넌트 근처에서 선언할 수 있게 되었고, `precedence` 속성으로 우선순위를 지정할 수 있다.
  - React가 이 우선순위를 바탕으로 `<link rel="stylesheet">`의 삽입 순서를 관리해주기 때문에 스타일 충돌을 덜 수 있다.
  - 더 흥미로운 점은 Suspense와의 연결이다. 어떤 UI가 특정 stylesheet에 의존한다면, React는 필요한 스타일이 준비되기 전에 그 UI를 성급하게 노출하지 않도록 조정한다.
  - 즉 React 19는 스타일 로딩도 단순 리소스 로딩이 아니라 렌더링 흐름의 일부로 다루기 시작했다고 느꼈다.

- `useImperativeHandle`
  - React는 기본적으로 선언형 UI를 지향하지만, 포커스 이동이나 스크롤 제어처럼 명령형 제어가 더 자연스러운 순간이 있다.
  - `useImperativeHandle`은 이럴 때 부모가 자식의 내부 DOM 전체를 직접 만지는 대신, 자식이 허용한 메서드만 외부로 노출하도록 도와준다.
  - 예를 들면 `focus()`, `scrollIntoView()` 같은 기능만 열어두고 내부 구현은 감출 수 있다.
  - React 19에서는 `ref`를 prop으로 받을 수 있게 되어서, 이 패턴도 이전보다 조금 더 자연스럽게 느껴진다.
  - 다만 이 훅은 어디까지나 예외적인 도구다. `isOpen` 같은 상태를 prop으로 표현할 수 있다면 ref 기반 명령형 제어보다 선언형 방식이 더 React답다.

- 동시성 기능 이해하기
  - React의 동시성 기능은 "코드 실행을 나중에 미룬다"기보다 "사용자 입력처럼 더 급한 업데이트를 먼저 처리할 수 있게 우선순위를 나눈다"는 쪽에 가깝다.
  - 그래서 동시성 관련 훅을 이해할 때는 비동기 처리 자체보다도, 어떤 업데이트를 덜 급한 것으로 볼지 생각하는 게 중요하다.

- `useTransition`
  - `useTransition`은 `[isPending, startTransition]`을 반환하고, `startTransition`으로 감싼 업데이트를 낮은 우선순위의 transition으로 표시한다.
  - 이 덕분에 탭 전환, 검색 결과 목록 갱신, 무거운 화면 전환처럼 비용이 큰 업데이트가 있더라도 입력이나 클릭 같은 상호작용은 더 즉각적으로 반응할 수 있다.
  - `isPending` 값으로는 "지금 transition이 진행 중인지"를 표시할 수 있어서, 사용자에게 로딩 피드백을 주기 좋다.
  - 중요한 점은 `startTransition` 안의 함수가 늦게 실행되는 것이 아니라, 그 안에서 발생한 상태 업데이트의 우선순위가 낮아진다는 것이다.
  - React 19에서는 transition 안에서 비동기 action을 다루는 흐름이 더 좋아졌고, pending 상태와 함께 데이터 변경 작업을 구성하기가 한결 자연스러워졌다.

- `useDeferredValue`
  - `useDeferredValue`는 상태를 직접 늦추는 훅이 아니라, 어떤 값의 "덜 급한 버전"을 하나 더 만들어주는 훅이라고 이해하면 편하다.
  - 예를 들어 입력창 값은 즉시 바꾸되, 그 값을 기반으로 렌더링되는 무거운 검색 결과 목록은 조금 늦게 따라오게 만들 수 있다.
  - 그래서 사용자는 타이핑이 버벅이지 않고, 결과 목록은 백그라운드에서 천천히 최신 값으로 따라붙는다.
  - Suspense와 함께 쓰면 더 인상적인데, 새 값으로 렌더링하는 도중 suspend가 발생해도 즉시 fallback으로 바꾸기보다 이전 값을 잠시 유지하면서 더 안정적인 경험을 줄 수 있다.
  - 결국 이 훅은 "최신성"보다 "반응성"이 더 중요한 화면에서 특히 유용하다.

- `useSyncExternalStore`
  - 이 훅은 React 바깥에서 관리되는 외부 스토어를 React 컴포넌트와 안전하게 연결하기 위한 표준 API다.
  - 브라우저 API의 `navigator.onLine` 같은 값이나, 서드파티 상태관리 라이브러리처럼 React 외부에서 변경되는 값을 구독할 때 적합하다.
  - `subscribe`로 변경을 감지하고, `getSnapshot`으로 현재 값을 읽는 구조가 핵심이다.
  - 예전에는 `useEffect`로 구독하고 정리하는 패턴을 직접 작성해야 했는데, `useSyncExternalStore`는 이 과정을 React 렌더링 모델에 맞게 표준화해준다.
  - 특히 여러 컴포넌트가 같은 외부 상태를 볼 때 일관된 스냅샷을 읽게 만들어 티어링 같은 문제를 줄이는 데 의미가 있다.
  - 다만 공식 문서에서도 말하듯, 가능하다면 React 내부 상태를 먼저 고려하고 정말 외부 스토어와 연결해야 할 때 이 훅을 쓰는 편이 좋다.
