- 이번 주차를 공부하면서 가장 크게 느낀 점은 React가 예전보다 훨씬 더 "렌더링 실패", "로딩 상태", "업데이트 우선순위"를 컴포넌트 바깥의 예외 상황이 아니라 렌더링 과정 자체의 일부로 다룬다는 점이었다. 예전에는 데이터를 가져오면 `isLoading`, `error`, `data`를 직접 나눠 관리하고, 예외는 `try/catch`나 개별 분기문으로 처리하는 방식이 익숙했다. 그런데 이번에 `ErrorBoundary`, `Suspense`, `use()`를 같이 보면서 React는 로딩이면 `<Suspense>`가, 에러면 `<ErrorBoundary>`가, 성공이면 컴포넌트가 그대로 렌더링되도록 책임을 분리하는 방향으로 발전해 왔다는 생각이 들었다. 특히 `use()`는 Promise를 읽다가 아직 준비되지 않았으면 suspend하고, 실패하면 에러를 전파한다는 점에서 비동기 처리조차 렌더링 규칙 안으로 끌어온 API처럼 느껴졌다.

- 또 하나 인상 깊었던 부분은 동시성 훅들이 단순히 "성능 최적화용 고급 기능"이 아니라, 사용자 경험을 설계하는 도구라는 점이었다. `useTransition`과 `useDeferredValue`를 보면 중요한 것은 연산 자체를 없애는 것이 아니라, 어떤 업데이트를 먼저 보여주고 어떤 업데이트는 잠시 늦춰도 되는지 구분하는 일이다. 예를 들어 입력창 값은 즉시 반응해야 하지만, 그 입력값으로 인해 렌더링되는 무거운 리스트나 검색 결과는 조금 늦게 보여줘도 괜찮다. React는 이런 상황에서 모든 상태를 동일한 중요도로 처리하지 않고, 사용자가 체감하는 반응성을 기준으로 우선순위를 조정할 수 있게 해준다. 이 부분을 보면서 프론트엔드 개발에서 중요한 것은 "최신 데이터만 빨리 보여주기"보다 "사용자가 답답하지 않게 느끼는 흐름을 만드는 것"이라는 점을 다시 느꼈다.

- 마지막으로 이번 주차는 React가 선언형 철학을 유지하면서도, 꼭 필요한 곳에서는 명령형 제어와 플랫폼 기능까지 품으려 한다는 점도 보여줬다. `useImperativeHandle`은 부모가 자식의 내부 구현을 모두 건드리는 대신 정말 필요한 메서드만 노출하게 만들고, `useSyncExternalStore`는 React 밖의 상태를 React의 렌더링 모델에 안전하게 연결하게 해준다. 여기에 React 19의 메타데이터 태그 처리나 stylesheet `precedence` 지원까지 보고 나니, React는 이제 단순히 UI만 그리는 라이브러리가 아니라 브라우저 환경 전체를 더 선언적으로 다루게 해주는 도구에 가까워지고 있다는 생각이 들었다. 결국 이번 주차에서 배운 내용은 "훅을 많이 아는 것"보다, 어떤 상태를 어떤 경계에서 다루고 어떤 경험을 사용자에게 보여줄지 설계하는 사고방식을 익히는 데 더 의미가 있었다.
